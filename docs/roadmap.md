# 로드맵 — Phase 정의와 완료 조건

현행 기준 문서 (노션 기획서와 동기 유지 — 변경 시 양쪽 모두 갱신).
세션 단위 task 문서는 **각 Phase 착수 시점에** `docs/tasks/`에 작성한다. 선제 상세화 금지 — 이유는 `docs/archive/` 참조.

## Phase 1 — MVP (진행 중, ~8/31)

**범위**: 코어 루프(AI 등록 → LIKE 검색 → 채팅) + React 코어 화면 5종 + ApplicationEvent 뼈대 + CI/ArchUnit + k6 도구 준비

**완료 조건 (DoD)**
- [ ] 코어 루프가 배포 환경에서 동작: 회원가입 → AI 도움 상품 등록 → 검색 → 채팅
- [ ] ModuleBoundaryTest가 CI에서 머지 게이트로 동작
- [ ] k6 Before 기준선 측정 완료 — 검색·상품 조회 p95를 `docs/measurements/baseline.md`에 기록
- [ ] `v0.1.0` 릴리스 (릴리스 노트에 Before 수치 포함)

## Phase 2 — 측정 기반 개선 (9월~, 공채 지원과 병행)

**진입 조건**: Phase 1 DoD 충족 (Before 기준선 없이는 착수 금지 — 비교쌍이 사라짐)

**작업 묶음 (순서)**
1. Redis 캐시 도입 (인기 상품 목록·상품 상세) → After 측정
2. ApplicationEvent → Kafka 전환 (`product.created`, 색인·알림 컨슈머 그룹 분리) → 전환 근거를 기획서 6번에 기록
3. LIKE → Elasticsearch 전환 + Kafka 색인 파이프라인 → After 측정
4. 알림 1~2종 구현 (Kafka 구독자)

**완료 조건 (DoD)**
- [ ] 검색 p95 가설(≤300ms @ 100VU) 달성 여부 판정 + Before 대비 개선율이 `docs/measurements/`에 기록됨
- [ ] 캐시 대상 조회의 히트율 측정값 기록 (가설: ≥80%)
- [ ] 장애 격리 실증: ES 컨테이너 중지 상태에서 상품 등록 API 정상 동작 확인 기록
- [ ] 기획서 6번(기술 선정 근거)의 Redis/Kafka/ES 항목이 추측이 아닌 실측 데이터로 채워짐
- [ ] `v0.2.0` 릴리스 (Before/After 비교표 포함)

## Phase 3 — 조건부 분리 (Phase 2 완료 후)

**진입 조건**: Phase 2 부하 데이터로 모듈별 병목 특성 식별 완료

**작업 묶음 (순서)**
1. MSA 전환 트리거 기준(기획서 6번) 대조 → 분리 대상 확정 (후보: 채팅, 검색) → ADR 작성
2. 대상 모듈 ID 참조 전환 → 서비스 분리 → docker-compose 멀티 서비스 + 게이트웨이/라우팅 구성
3. 분리 전후 비교 측정 (성능 + 배포·장애 독립성)

**완료 조건 (DoD)**
- [ ] `docs/adr/`에 분리 결정 ADR 1건 이상 — **분리하지 않은 모듈의 사유 포함 필수**
- [ ] 분리된 서비스의 독립성 실증: 모놀리스와 분리 서비스 중 한쪽 중단 시 다른 쪽 핵심 기능 유지 확인 기록
- [ ] 분리 전후 비교표 (지연·처리량·운영 복잡도 변화)
- [ ] `v0.3.0` 릴리스

## 확장 트랙 — AI 거래 대리인 (Phase 2~3과 병행 가능, 독립 트랙)

**L2 — 가격 타당성 리포트**
- 진입 조건: MVP 등록 플로우 안정 + 매물 데이터 유무 판단 (초기: 모델 일반 지식 기반)
- DoD: 등록 시 가격 제안에 근거(비교 기준) 표시, 골든셋 N건으로 제안 품질 측정 기록

**L3 — 채팅 응답 초안 보조**
- DoD: 초안 제안 기능 동작 + **전송은 반드시 사용자 승인** (자동 전송 경로가 코드상 존재하지 않음을 테스트로 증명), AI 관여 사실이 UI에 표시

**L4 — 위임 기반 협상·일정 대리**
- DoD: mandate(가격 하한·시간대·장소)가 시스템 강제임을 테스트로 증명 (LLM 출력이 하한 미달 제안 시 코드 레벨 거절), 확정 행위는 human-in-the-loop 승인 필수, 상대방에게 AI 대리인 고지 배지 표시

**결제 (확장 트랙 마지막)**
- 범위: Toss 테스트 모드, 결제 승인·취소, 멱등성 키, 결제 상태 전이
- DoD: 동일 주문 중복 승인 0건이 동시성 테스트로 증명됨, 상태 전이가 상태 머신으로 정의되고 비정상 전이가 테스트로 차단됨

## 공통 규칙

- 각 Phase 종료 시 회고(KPT)를 `docs/`에 남긴다
- Phase DoD의 체크박스가 전부 채워지기 전에는 다음 Phase의 구현 작업을 시작하지 않는다 (확장 트랙은 예외 — 병행 가능)
- 일정 지연 시 리스크 컷 정책(기획서 7번)에 따라 범위를 줄이고, 줄인 결정을 기록한다
