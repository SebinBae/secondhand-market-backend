# 3.3 채팅 및 실시간 알림

- 채팅방 생성 (특정 상품 & 구매자 기준, 1:1, 동일 상품-동일 구매자는 1개만 생성)
- 채팅방 목록 조회 (마지막 메시지 시간 순)
- 채팅 메시지 송수신 (STOMP WebSocket)
- 채팅 메시지 목록 조회 (REST, createdAt 오름차순 + offset 페이지네이션)
- 메시지 읽음 확인 및 타이핑 중 표시 **(신규)**
- Redis Pub/Sub 기반 메시지 브로드캐스팅 — 멀티 인스턴스 대응 **(신규)**
- 거래 상태 변경 시 Kafka로 이벤트 발행 → 알림 서비스가 구독해 비동기 처리 **(신규)**
- 실시간 알림 — 접속 중이면 SSE, 미접속/모바일이면 FCM 푸시 **(신규)**
- 안 읽은 메시지/알림 개수 뱃지 표시 **(신규)**
- 알림 수신 설정 (채팅/거래상태 알림 on·off 토글) **(신규)**
- WebSocket 핸드셰이크 허용 Origin을 REST API CORS 정책과 동일하게 제한 (* → 운영 도메인) **(신규)**
- 채팅 메시지 목록 조회를 커서 기반 페이지네이션으로 전환 — 채팅방 진입 시 최신 메시지 N개를 먼저 조회하고, 위로 스크롤하면 마지막으로 받은 메시지의 createdAt/id를 커서로 그 이전 메시지를 조회 **(신규)**

## 구현 디테일

PK가 랜덤 UUID라 id 단독으로는 정렬 기준이 될 수 없어 createdAt을 1차 기준, id를 동률 처리용 보조 커서로 사용. JPQL/QueryDSL은 튜플 비교((created_at, id) <)를 지원하지 않아 `createdAt < cursor OR (createdAt = cursor AND id < cursorId)` 형태로 풀어서 구현(ProductQueryRepositoryImpl과 동일한 QueryDSL 패턴 재사용).

(chat_room_id, created_at, id) 복합 인덱스를 추가하지 않으면 방의 메시지가 많아질 때 정렬 비용이 커짐. 응답도 Spring Data의 `Page<T>`(offset 기반, totalElements 계산 포함)에서 `List<ChatMessageResponse> + nextCursor + hasNext` 형태의 커스텀 DTO로 전환 — 끝없이 쌓이는 채팅 로그에서는 전체 개수 계산이 비용 대비 의미가 적음.

## 기술 선택 이유 (신규)

| 기술 | 선택 이유 |
| --- | --- |
| Redis Pub/Sub (WebSocket 브로드캐스팅) | 기본 STOMP 메시지 브로커는 단일 인스턴스 메모리 안에서만 동작해, 서버를 2대 이상으로 확장하면 다른 인스턴스 사용자에게 메시지가 전달되지 않음. Redis Pub/Sub을 메시지 릴레이로 두면 모든 인스턴스가 같은 채널을 구독해 수평 확장이 가능한 실시간 아키텍처를 구현 |
| Kafka (이벤트 기반 비동기 처리) | 결제완료 같은 이벤트 발생 시 알림 발송·검색 인덱싱·정산 처리를 동기로 처리하면 응답 지연이 커지고 한 작업 실패가 전체 롤백 위험으로 이어짐. 이벤트 발행 후 각 서비스가 독립적으로 구독·처리하게 해 느슨한 결합과 장애 격리를 동시에 확보 |
| SSE (서버 푸시 알림) | 알림은 폴링 없이 발생 시점에 바로 전달하는 게 자연스러움. WebSocket보다 가볍고(단방향), 서버→클라이언트 단방향 푸시에 적합 |
| FCM (모바일/미접속 푸시) | 앱을 닫아두거나 탭이 꺼져 있으면 SSE 연결이 끊기므로, OS 레벨 푸시로 그 상황까지 커버 |
| 읽음 확인 (메시지 상태 필드) | 단순한 변경이지만, 거래 협의에서 "상대가 봤는지" 여부가 신뢰 문제와 직결되어 UX 임팩트가 큼 |
| 커서 기반 페이지네이션 (메시지 조회) | 기존 offset(page/size) 방식은 최신 메시지를 보려면 전체 개수를 먼저 알아야 하고, 메시지가 계속 쌓이는 채팅방에서는 같은 페이지 요청이 시점에 따라 다른 결과를 반환할 수 있음(오프셋 시프트). createdAt+id 복합 커서로 "이 메시지보다 이전 N개"를 조회하면 안정적으로 무한 스크롤을 구현할 수 있음 |
| 복합 인덱스 (chat_room_id, created_at, id) | 현재 엔티티엔 chat_room_id FK 기본 인덱스만 있고 created_at과 결합된 인덱스가 없음. 커서 쿼리가 매번 정렬용 풀스캔/임시 정렬을 하지 않도록 조회 패턴과 동일한 순서의 복합 인덱스를 추가 |
| QueryDSL 기반 keyset 비교 | JPQL/QueryDSL은 (created_at, id) < 같은 튜플 비교를 지원하지 않아, `createdAt.lt(cursor).or(createdAt.eq(cursor).and(id.lt(cursorId)))` 형태로 직접 풀어서 구현. 이미 상품 검색(ProductQueryRepositoryImpl)에 쓰던 QueryDSL을 그대로 재사용해 새 라이브러리 추가 없이 일관된 패턴 유지 |
