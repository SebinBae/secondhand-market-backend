# 중고거래 마켓 플랫폼 — 서비스 개요

기존 secondhand-market-backend 프로젝트를 기반으로, 다양한 기술 스택을 적용해 신입 개발자 포트폴리오 경쟁력을 높이기 위한 확장 기획서입니다.

## 1. 서비스 개요

| 항목 | 내용 |
| --- | --- |
| 서비스명 | 중고거래 마켓 (가칭) |
| 한 줄 소개 | AI 등록 도우미와 안전거래 시스템을 갖춘 실시간 중고거래 플랫폼 |
| 도메인/비즈니스 | 중고 물품 등록·검색·채팅·거래 (당근마켓 스타일) — 변경 없음 |
| 타겟 사용자 | 동네(생활권) 단위 직거래를 선호하는 사용자. 도보·대중교통으로 직접 만나 거래 가능한 거리 내 이웃 간 매칭이 핵심 가치 — 당근마켓과 동일한 하이퍼로컬 모델 |
| 예상 트래픽/운영 규모 | 실 사용자 트래픽을 목표 수치로 설정하지 않음. 하이퍼로컬 서비스는 동네 단위로 거래·채팅이 잘게 나뉘어 동시에 발생하는 구조라, 트래픽 총량보다 **여러 동네의 트랜잭션이 독립적으로 동시 처리되는 상황을 견디는 아키텍처**인지를 증명하는 데 집중 (nfr.md 4.7 검증 계획 참고) |
| 핵심 차별점 | LangGraph 기반 AI 상품 등록 에이전트, 자체 구현 안전거래(에스크로 패턴), 멀티 인스턴스 대응 실시간 아키텍처 |
| 목표 | 신입 AI 백엔드/풀스택 개발자 채용 포트폴리오 |

**타겟이 아키텍처에 미치는 영향**: 하이퍼로컬 구조에서는 "동네 A의 거래"와 "동네 B의 거래"가 서로 독립적이면서도 동시에 일어난다. 이는 채팅방·알림이 다수의 작은 단위로 분산되는 부하 패턴을 만들며, Redis Pub/Sub 멀티 인스턴스 브로드캐스팅과 Kafka 기반 비동기 색인 설계가 "트래픽 총량이 적어도 왜 의미 있는 설계인지"를 뒷받침하는 근거가 된다.

## 2. 전체 사용자 시나리오

개별 기능이 아니라, 한 거래가 처음부터 끝까지 어떻게 흘러가는지를 통해 전체 기능이 어떻게 맞물리는지 보여주는 시나리오입니다.

**등장인물**: 판매자 A, 구매자 B

- **상품 등록 (AI 에이전트)**: A는 안 쓰는 카메라를 팔려고 사진 한 장과 "소니 카메라 팝니다, 상태 괜찮아요" 라는 한 줄 설명만 입력한다. AI 상품 등록 도우미(LangGraph)가 이미지를 분석해 카테고리를 분류하고, 유사 상품 임베딩 검색(Supabase pgvector)으로 시세를 추정해 가격을 추천하며, 제목/설명을 자동으로 채워준다. A는 결과를 확인하고 등록 버튼만 누른다.

- **검색 (Elasticsearch)**: B는 "소니 카메라"를 검색한다. Elasticsearch + Nori 분석기가 오타와 표현 변형까지 잡아 A의 상품을 찾아준다. 상품 상세 페이지에서는 임베딩 유사도 기반으로 "비슷한 카메라" 추천도 함께 보여준다.

- **채팅 (실시간 고도화)**: B는 채팅으로 A에게 가격 협의를 요청한다. A가 메시지를 읽었는지 읽음 표시로 확인할 수 있고, 답장을 작성 중이면 타이핑 표시가 뜬다. 두 서버 인스턴스에 각각 연결되어 있어도 Redis Pub/Sub을 통해 메시지가 정상적으로 전달된다.

- **결제 (안전거래)**: 가격 협의가 끝나고 B가 결제를 진행한다. 토스페이먼츠 테스트 모드로 결제가 승인되지만, 정산은 즉시 이루어지지 않고 PENDING 상태로 보류된다.

- **실시간 알림**: 결제 완료 이벤트가 Kafka로 발행되고, 알림 서비스가 이를 구독해 A에게 "결제가 완료되었습니다"라는 알림을 보낸다. A가 접속 중이면 SSE로, 앱을 닫아두었다면 FCM 푸시로 전달된다.

- **구매 확정 및 정산**: 물건을 받은 B가 "구매 확정" 버튼을 누르면 정산 상태가 CONFIRMED로 바뀌고 A에게 정산 알림이 간다. B가 확정을 누르지 않고 7일이 지나면 스케줄러가 자동으로 확정 처리한다.

이 시나리오 하나로 AI 에이전트 → 검색/추천 → 실시간 채팅 → 결제/안전거래 → 실시간 알림 → 정산까지 전체 신규 기능이 한 흐름 안에서 자연스럽게 동작하는 것을 보여줄 수 있습니다.

## 3. 기능 명세 안내

기존 기능과 신규 기능을 별도 챕터로 나누지 않고 **기능 카테고리별로 통합**했습니다. 각 도메인별 상세 명세는 아래 파일을 참고하세요.

- `01-auth.md` — 회원/인증
- `02-product.md` — 상품 관리
- `03-chat-notification.md` — 채팅 및 실시간 알림
- `04-payment.md` — 결제 및 안전거래
- `05-search.md` — 검색 및 추천
- `06-frontend.md` — React 기반 웹 프론트엔드
- `07-admin.md` — 관리자 페이지
- `nfr.md` — 비기능 요구사항 (NFR)
- `architecture.md` — 전체 아키텍처, 로드맵, 개발 프로세스, 스코프 아웃 항목

**코드 점검에서 발견된 이슈** (각 도메인 파일의 (신규) 항목으로 반영됨):

- JWT는 Access Token만 발급되고 Refresh Token 발급/회전 로직이 없어, Access Token 만료(20분)마다 재로그인이 필요함
- AuthController에 로그아웃 API가 존재하지 않음 (docs/README.md에는 기존 기능으로 적혀 있으나 실제 코드와 불일치)
- JwtAuthenticationFilter, JwtChannelInterceptor가 Authorization 헤더와 토큰 원문을 log.info로 그대로 출력 — 로그 유출 시 토큰 탈취 경로가 될 수 있음
- WebSocket 핸드셰이크는 setAllowedOriginPatterns("*")로 모든 Origin을 허용해, REST API CORS 정책(localhost:3000만 허용)과 어긋남
- GlobalExceptionHandler가 BusinessException만 처리해, @Valid 검증 실패나 토큰의 사용자가 DB에 없는 경우(orElseThrow()) 등은 처리되지 않은 예외로 새어나갈 수 있음
- 본인 리소스 검증은 서비스 레이어의 수동 if 비교로 구현되어 있음
- ChatMessageController의 메시지 목록 조회가 createdAt 오름차순 + offset(page/size) 방식으로 구현되어 있음 — 오프셋 시프트 문제로 불안정함
