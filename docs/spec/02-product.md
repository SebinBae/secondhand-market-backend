# 3.2 상품 관리

- 상품 등록 (제목/가격/설명/카테고리)
- 상품 목록 조회 (최신순/가격순 정렬, QueryDSL 동적 검색)
- 상품 수정/삭제 (작성자 본인만 가능)
- 상품명 기반 검색
- 거래 상태 변경 (판매중/거래완료)
- 상품 이미지 업로드 (상품 1개당 다중 이미지, Supabase Storage) **(신규)**
- 관심상품 찜하기 / 내 찜 목록 조회 **(신규)**
- AI 상품 등록 도우미 — 사진 1~2장 + 한 줄 설명만 입력하면 카테고리 분류 → 가격 추천 → 제목/설명 자동 생성 **(신규)**

다중 이미지를 지원하기 위해 ProductEntity에 이미지 필드를 직접 두지 않고, ProductImage(상품 1 : 이미지 N) 엔티티를 분리합니다. AI 등록 도우미는 이 중 대표 이미지를 비전 모델 입력으로 사용합니다.

## LangGraph 노드 구조

Intake(사진+한줄설명 입력) → Vision(이미지 분석, 상태/종류/브랜드 추정) → Category(카테고리 분류) → PriceEstimation(임베딩 유사도 검색 + 가격 추천) → Description(제목/설명 자동 생성).

Category 노드의 분류 신뢰도가 낮으면 Vision 노드로 되돌아가는 재시도 분기 하나만 둡니다 — 분기를 더 늘리면 그래프가 복잡해져 면접에서 설명하기 어려워지므로 의도적으로 단순하게 유지합니다.

## 기술 선택 이유 (신규)

| 기술 | 선택 이유 |
| --- | --- |
| Supabase Storage (이미지 저장) | 이미 AI 도우미용 pgvector 때문에 Supabase 프로젝트를 운영하므로, 별도 오브젝트 스토리지(S3/R2) 계정·버킷을 새로 만들지 않고 같은 프로젝트 안에서 Storage 기능만 켜서 사용. RLS(Row Level Security)로 pgvector와 동일한 정책 체계로 접근 제어 가능 |
| LangGraph (멀티 에이전트) | 이미지 분석 → 카테고리 분류 → 유사상품 검색 → 가격 추정 → 텍스트 생성처럼 순차적이면서도 조건 분기(예: 분류 실패 시 재질문)가 필요함. 단일 프롬프트로는 단계별 검증/재시도가 어렵고 책임이 섞여 디버깅이 힘듦. 상태 기반 그래프 구조로 각 단계를 노드로 분리해, 특정 단계만 실패해도 그 노드만 재시도 가능하게 설계 |
| 비전 모델 (이미지 분석) | 텍스트로 카테고리/특징을 일일이 입력하게 하면 진입장벽이 높음. 사진 한 장으로 상태/종류/브랜드 추정 정보를 추출해 입력 부담을 최소화 |
| 임베딩 + 유사도 검색 (가격 추천 노드) | "적정 가격"은 정답이 없는 문제라 LLM이 그냥 추측하면 신뢰도가 떨어짐. 기존 거래 데이터에서 유사 상품을 임베딩 기반으로 검색해, 실제 거래 가격대를 근거로 추천해 환각(hallucination) 위험을 줄임 |
| FastAPI (Python AI 서비스 분리) | LangGraph/임베딩 등 AI 생태계는 Python이 표준. Spring Boot(도메인/인증/결제)와 AI 로직을 REST로 분리해 각 서비스가 자기 강점 언어를 쓰도록 책임을 나눔 |
| Supabase (pgvector) | 벡터 저장/유사도 검색을 위해 별도 벡터 DB를 직접 운영하면 인프라 관리 부담이 커짐. Postgres 기반이라 SQL로 벡터 검색을 표준화할 수 있고, 무료 티어로 빠르게 구축 가능 |
