# 3.6 React 기반 웹 프론트엔드 — 전체 신규 (신규)

## 문제 상황

지금까지는 백엔드 API만 존재해, 실제로 써볼 수 있는 화면이 없다. API 명세만으로는 서비스를 이해하기 어렵고, 백엔드가 실제로 잘 동작하는지 증명하는 가장 확실한 방법은 동작하는 클라이언트를 붙여보는 것이다.

## 해결

지금까지 설계한 모든 기능(상품/채팅/결제/실시간 알림/AI 에이전트/검색)을 사용할 수 있는 React SPA를 구축한다.

## 기능 상세

- 상품 목록/상세/등록(AI 도우미 포함)/수정 페이지 + 찜하기
- 실시간 채팅 UI (WebSocket 연결, 읽음 표시, 타이핑 인디케이터)
- 결제 위젯 연동 페이지 + 구매확정/거래내역 페이지 + 거래 후기 작성
- 실시간 알림 UI (SSE 구독, 알림 뱃지/토스트, 알림 수신 설정)
- 검색/자동완성 UI + 비슷한 상품 추천 섹션
- 로그인/회원가입 (소셜 로그인 포함, JWT 토큰 관리, Refresh Token 갱신 흐름, 비밀번호 찾기)
- 마이페이지 (내 상품, 찜 목록, 내 거래내역, 정산 현황, 회원정보 수정, 회원 탈퇴)
- 신고/차단 UI (상품·채팅 화면에서 진입)
- 동네 인증 UI (GPS 권한 요청 → 인증 결과 표시, 마이페이지에서 재인증)

## 페이지 라우팅 구조

`/products`, `/products/:id`, `/products/new`, `/products/:id/edit`, `/chat`, `/chat/:roomId`, `/search`, `/mypage`, `/mypage/products`, `/mypage/orders`, `/mypage/wishlist`, `/mypage/settings`, `/login`, `/signup`, `/admin/*`(역할 기반 보호 라우트 — `/api/users/me`의 role이 ADMIN이 아니면 리다이렉트)

## 기술 선택 이유

| 기술 | 선택 이유 |
| --- | --- |
| React + TanStack Query | 서버 상태가 자주 바뀌는 화면이 많아 캐싱·리페칭·로딩/에러 상태를 직접 구현하면 보일러플레이트가 많아짐. 서버 상태 관리를 분리해 클라이언트 상태(Zustand)와 명확히 구분되는 구조를 구현 |
| Zustand (클라이언트 상태) | 로그인 정보, UI 상태 같은 전역 클라이언트 상태는 서버 캐싱이 필요 없어 Redux 대비 보일러플레이트가 적은 라이브러리로 가볍게 관리 |
| WebSocket/STOMP + SSE 클라이언트 공존 | 채팅은 양방향 실시간이 필요하지만 알림은 서버→클라이언트 단방향이라 SSE가 더 가벼움. 백엔드 설계 의도(양방향 vs 단방향)에 맞춰 두 가지 연결 방식을 적합한 곳에 사용 |
| Refresh Token 자동 갱신 (Axios Interceptor) | 백엔드에 Refresh Token 발급/회전을 추가하므로(01-auth.md 참고), 프론트는 Access Token 만료(401) 응답을 인터셉터로 잡아 자동 재발급받는 흐름을 구현. 백엔드 Refresh Token 구조가 먼저 갖춰져야 동작 가능 |

**학습 메모**: TanStack Query/Zustand는 구현 단계에서 기본 개념(서버 상태 vs 클라이언트 상태, 캐시 무효화, selector 패턴 등)부터 정리 후 진행.
