# 4. 비기능 요구사항 (NFR)

## 4.1 확장성 (Scalability)

- WebSocket 메시지는 Redis Pub/Sub으로 브로드캐스팅해 다중 인스턴스에서도 메시지 수신 가능
- 이벤트 기반 비동기 처리(Kafka)로 결제완료/상품등록 후속 작업(알림, 검색 색인)을 메인 트랜잭션과 분리해 컨슈머만 스케일아웃 가능
- 모듈러 모놀리식 구조 유지 (MSA 전환은 트래픽 증가 시 단계적으로 검토)

## 4.2 일관성/정합성 (Consistency)

- 결제 웹훅 멱등성 처리 (orderId 기반 중복 처리 방지)
- 정산 상태(PENDING → CONFIRMED) 동시성 제어 (비관적 락 / Redis 분산락)
- MySQL ↔ Elasticsearch ↔ Supabase 간 최종 일관성(eventual consistency) — Kafka 이벤트로 비동기 동기화, 실패 시 재시도/DLQ 전략 필요

## 4.3 성능

- N+1 문제 예방 (fetch join / @EntityGraph / @BatchSize) — 현재는 응답 DTO가 연관 엔티티의 ID만 노출해 N+1이 발생하지 않지만, 판매자 닉네임 등 연관 필드를 추가 노출하는 순간 발생할 수 있어 사전에 대응
- 검색은 MySQL LIKE 대신 Elasticsearch로 분리해 응답 속도 개선
- 자주 조회되는 데이터(상품 목록, 인기 검색어)는 Redis 캐싱

## 4.4 보안

아래 항목은 업로드한 코드(secondhand-market-backend) 점검 결과를 반영해 실제 구현과 맞게 수정했습니다.

- Spring Security + JWT — 기존 Access Token(BCrypt 비밀번호 해싱 포함)에 Refresh Token 발급/회전을 추가하고, Refresh Token은 HttpOnly 쿠키로 전달 (01-auth.md)
- 인가는 컨트롤러 레벨 `@PreAuthorize("isAuthenticated()")` + 서비스 레이어의 리소스 소유자 직접 비교로 구현 — **현재 방식을 유지하기로 결정**. SpEL 기반 @PreAuthorize로의 전환을 검토했으나, 현재 도메인 복잡도에서는 가독성 이득이 크지 않고 디버깅도 수동 if 비교가 더 직관적이라 판단해 의도적으로 유지
- 인증 관련 로그에서 Authorization 헤더·토큰 원문 노출 제거 (현재 log.info로 토큰이 그대로 출력되는 문제 수정)
- WebSocket 핸드셰이크 허용 Origin을 REST API CORS 정책과 통일 (03-chat-notification.md)
- 인증/검증 실패 예외 처리 표준화 — BusinessException 외 @Valid 검증 실패, 토큰 사용자 미존재 등도 공통 응답 포맷(ApiResponse)으로 처리
- 결제 웹훅 요청 서명 검증으로 위조 요청 차단
- 이미지 업로드 입력 검증 — 크기/형식(MIME) 제한으로 비용 폭증 및 악성 파일 업로드 방지 (AI 에이전트 입력, 상품 이미지 공통 적용)

## 4.5 가용성/장애 격리

- 결제/검색/알림은 Kafka로 비동기 분리해, 장애가 핵심 거래 기능(상품 등록/조회)에 영향 주지 않도록 격리
- AI 에이전트(Python/FastAPI) 장애 시에도 수동 상품 등록은 계속 가능 (AI는 보조 기능으로 설계)

## 4.6 모니터링/운영

- Prometheus + Grafana로 시각화. 우선 지표 5가지: ① API 응답시간(p50/p95/p99) ② 결제 성공률 ③ Kafka 컨슈머 lag ④ AI 에이전트 응답시간/실패율 ⑤ Redis 캐시 히트율 — 그 외 지표는 필요 시 추가

## 4.7 기술 선택 트레이드오프 및 검증 계획

포트폴리오 프로젝트는 실제 트래픽이 거의 없기 때문에, "트래픽이 많아서"라는 근거만으로는 Redis/Kafka 같은 이벤트 기반 도구 도입을 정당화하기 어렵다. 대신 **응답 지연 분리 / 장애 격리** 같은 트래픽 규모와 무관한 근거를 중심으로 설계하고, 실제로 효과를 측정해 증명한다.

### 가벼운 대안과의 비교

| 상황 | 가벼운 대안 | 선택한 기술 | 선택 기준 |
| --- | --- | --- | --- |
| 결제 완료 후 알림/색인 비동기 처리 | Spring @Async + @TransactionalEventListener (단일 JVM 내 비동기) | Kafka | 같은 JVM 안에서만 처리하면 @Async로도 충분함. 다만 **여러 서비스(알림, 검색 색인, AI 추천 갱신)가 같은 이벤트를 독립적으로 구독**해야 하고, 컨슈머별 재시도/순서 보장이 필요해 메시지 브로커가 더 적합하다고 판단 |
| WebSocket 메시지 멀티 인스턴스 전달 | 단일 인스턴스로만 운영 (확장 안 함) | Redis Pub/Sub | 실제 운영 트래픽은 단일 인스턴스로 충분할 수 있으나, **수평 확장 가능한 실시간 아키텍처 설계 역량**을 보여주기 위해 의도적으로 도입. "지금 필요해서"가 아니라 "이 패턴을 검증해보기 위해"라는 점을 명확히 함 |

### 검증 계획 (말로만 주장하지 않고 실측)

**응답 지연 비교 측정**
- Kafka 도입 전(동기 처리)과 도입 후(비동기 처리)의 결제 승인 API 응답시간을 각각 측정해 비교
- 측정 도구: k6 또는 Locust로 간단한 부하 시나리오 작성 (예: 동시 요청 N건)
- 결과를 기획서/README에 수치로 기록 (예: "동기 처리 시 평균 OOms → 비동기 분리 후 OOms")

**멀티 인스턴스 동작 실증**
- docker-compose로 Spring Boot 앱 2개 인스턴스 + Nginx 로드밸런서 구성
- 인스턴스 A에 연결된 사용자와 인스턴스 B에 연결된 사용자가 같은 채팅방에서 실제로 메시지를 주고받는지 시연 (Redis Pub/Sub 없이 했을 때 실패하는 것도 함께 보여주면 더 설득력 있음)

**장애 격리 실증**
- Elasticsearch 컨테이너를 강제로 중지시킨 상태에서 상품 등록 API가 정상 동작하는지 확인 (검색 색인만 실패하고 핵심 트랜잭션은 영향받지 않음을 증명)

이 측정/실증 결과는 README 또는 별도 기술 문서에 "Before/After" 형태로 정리해, 면접에서 "왜 이 기술을 썼나요?"에 추측이 아닌 데이터로 답할 수 있도록 한다.

### Kafka 토픽 구조

네이밍은 도메인.이벤트 형식. 컨슈머 그룹은 **용도별로 분리**해, 한쪽(예: 통계)이 재처리되어도 다른 쪽(예: 알림)이 영향받지 않도록 한다.

| 토픽 | 발행 시점 | 구독 컨슈머 |
| --- | --- | --- |
| product.changed | 상품 등록/수정/삭제 | ES 색인 컨슈머, AI 임베딩 갱신 컨슈머, 통계 컨슈머 |
| payment.completed | 결제 승인 완료 | 정산 PENDING 처리 컨슈머, 알림 컨슈머, 통계 컨슈머 |
| trade.status-changed | 구매확정/취소 등 거래 상태 변경 | 알림 컨슈머, 통계 컨슈머 |
| user.signup | 회원가입 완료 | 통계 컨슈머 |
| product.index.dlq | ES 색인 3회 재시도 실패 | (모니터링 대상, 수동/배치 재처리) |
