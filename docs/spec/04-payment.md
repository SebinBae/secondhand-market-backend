# 3.4 결제 및 안전거래 — 전체 신규 (신규)

## 문제 상황

중고거래는 비대면 거래 시 사기 위험이 크다. 구매자는 "돈을 보냈는데 물건을 안 보내면 어쩌지", 판매자는 "물건을 보냈는데 입금을 안 하면 어쩌지" 하는 신뢰 문제가 거래 포기의 주요 원인이다.

**참고**: 토스페이먼츠의 정식 에스크로 API는 사업자 별도 계약이 필요해 개인 포트폴리오로는 사용이 어렵다. 대신 일반 결제 + 자체 정산 보류 로직으로 안전거래 메커니즘을 직접 구현한다.

## 해결

토스페이먼츠로 결제를 받되, 판매자에게 즉시 정산하지 않고 구매자가 "구매 확정"을 눌러야 정산되는 보류 구조를 만들어 신뢰 문제를 해결한다.

## 기능 상세

- 결제위젯 연동 (카드, 계좌이체, 가상계좌) — **테스트 모드**로 운영 (실 결제 발생 없음)
- 결제 승인 API 호출 + 웹훅으로 결제 상태 수신
- 결제 완료 시 정산은 보류 상태(PENDING)로 기록
- 구매자가 물건 수령 후 "구매 확정" 클릭 → 판매자 정산 처리(CONFIRMED)
- 일정 기간(예: 7일) 내 확정/취소 없으면 스케줄러가 자동 확정
- 구매 취소/환불 요청 시 결제 취소 API 호출
- 구매/판매 내역 및 정산현황 조회 (마이페이지 연동)
- 구매확정 완료 건당 1회, 거래 후기 작성 — 선택형 태그(예: "시간 약속을 잘 지켜요", "친절해요", "응답이 빨라요") 방식, 누적 결과를 매너 점수로 환산해 프로필에 공개

## 기술 선택 이유

| 기술 | 선택 이유 |
| --- | --- |
| 토스페이먼츠 결제위젯 | 자체 PG 계약·카드사 심사 없이 표준화된 결제 UI/API를 빠르게 붙일 수 있고, 실제 서비스에서도 널리 쓰이는 결제 연동 경험을 보여줄 수 있음 |
| 웹훅 + 멱등성 처리 | 결제 승인은 콜백 외 웹훅으로도 수신되는데, 네트워크 재시도로 같은 이벤트가 중복 수신될 수 있음. orderId 기반 멱등키로 중복 처리를 방지해 데이터 정합성을 보장 |
| 정산 보류 상태 모델링 (PENDING → CONFIRMED) | 결제 완료 = 정산 완료가 아니라는 점을 도메인에 명확히 반영. 상태를 분리해야 "구매확정 전 취소", "자동확정 스케줄러" 같은 비즈니스 룰을 안전하게 구현할 수 있음 |
| 스케줄러 (자동 구매확정) | 구매자가 확정을 누르지 않고 방치하면 판매자가 영원히 정산받지 못하는 문제가 생김. 일정 기간 후 자동 확정 배치로 양쪽 모두를 보호 |
| 비관적 락 / Redis 분산락 | "구매확정"과 "환불요청"이 동시에 들어오면 같은 주문이 두 가지 최종 상태로 갈 수 있음. 동시성 제어로 한쪽만 성공하도록 보장 |
| 거래 후기 — 태그형 + 공개 | 자유 텍스트 후기는 모더레이션 부담과 악용(보복성 후기) 위험이 큼. 미리 정의된 태그 선택형으로 설계해 신뢰 신호는 유지하면서 운영 부담을 줄이고, 매너 점수로 집계해 공개해야 상대방의 구매 결정에 실제로 도움이 되는 정보가 됨 |
