# 3.7 관리자 페이지 — 전체 신규 (신규)

## 문제 상황

신고/차단(01-auth.md)을 만들면서 "누적 신고 시 운영자 검토 대상으로 플래그"라고 설계했는데, 정작 그걸 검토·처리하는 화면이 없다. 결제(04-payment.md)의 정산 보류 현황, 가입자·거래 통계 같은 운영 데이터도 지금은 DB를 직접 조회하지 않으면 볼 방법이 없다.

## 해결

화면 성격이 다른 두 영역을 분리해서 구현한다.

- **운영 처리 화면** (신고/차단 처리, 사용자 관리, 상품 관리) — 목록 + 처리 버튼 형태의 CRUD라 React 관리자 화면으로 구현
- **통계 대시보드** — 가입자/거래 추이 같은 시각화가 필요한 영역이라 Python 시각화 스택(Streamlit + Plotly)으로 별도 구현, 백엔드(Java)와는 다른 스택을 의도적으로 선택해 차별점으로 사용

## 기능 상세

- 신고/차단 처리 — 누적 신고 검토, 사용자 정지/차단 (React, `@PreAuthorize("hasRole('ADMIN')")`)
- 사용자 관리 — 회원 목록 조회, 강제 탈퇴/정지 (React)
- 상품 관리 — 신고된 상품 검토/강제 삭제 (React)
- 결제/정산 모니터링 — 정산 보류 현황, 환불 처리 (React)
- 통계 대시보드 — 가입자 수, 거래 건수, 카테고리별 인기 상품 추이 시각화 (Streamlit + Plotly)
- 통계 전용 Kafka 컨슈머 (Python) — 결제완료/상품등록/회원가입 이벤트를 구독해 집계 테이블에 적재, Streamlit은 이 집계 테이블만 조회
- 관리자 인증 — Streamlit/React 모두 기존 `/api/auth/login` 재사용, 로그인 후 `GET /api/users/me`로 role 확인해 관리자 화면 노출 여부 판단
- 관리자 액션 감사로그 — 신고처리/정지/환불 등 주요 액션을 "누가·언제·무엇을" 별도 테이블에 기록
- ADMIN role의 토큰은 Refresh Token을 발급하지 않음 — Access Token(20분) 만료 시 재로그인 필요 (고권한 계정의 토큰 노출 시간을 최소화)

## 기술 선택 이유

| 기술 | 선택 이유 |
| --- | --- |
| Streamlit + Plotly (통계 대시보드) | 데이터 → 차트 → 필터 화면을 빠르게 완성할 수 있고, 데이터 시각화 포트폴리오에서 가장 흔히 쓰는 도구라 면접관도 바로 알아봄. 고객용 앱(React)과 내부 운영툴(Python)을 다른 스택으로 가져가는 실제 스타트업 패턴을 그대로 반영 |
| React 관리자 화면 (CRUD 영역만) | 신고/차단·사용자·상품 관리는 결국 목록+처리버튼이라, Streamlit/Dash로 만들면 오히려 부자연스러움. 시각화가 필요 없는 CRUD는 기존 React 스택을 그대로 사용 |
| Kafka 통계 컨슈머 + 집계 테이블 | 통계 화면이 매번 운영 DB(MySQL)에 무거운 집계 쿼리를 날리면 핵심 트랜잭션에 부담을 줌. 이미 깔아둔 Kafka를 재사용해 이벤트 기반으로 집계 테이블에 미리 적재하고, Streamlit은 그 결과만 조회 — 새 인프라를 늘리지 않고 기존 자산을 재사용 |
| GET /api/users/me 기반 role 확인 (시크릿 비공유) | JWT가 HS256 대칭키라 Streamlit이 직접 디코드하면 서명 시크릿을 Python 환경에도 복제해야 함. 검증은 Spring Boot 한 곳에서만 유지하고 결과(role)만 받아 노출 표면을 늘리지 않음 |
| 회원가입 시 role 강제 고정 | 클라이언트가 role을 지정할 수 있게 두면 누구나 ADMIN으로 가입하는 권한 상승 취약점이 생김 |
| 관리자 액션 감사로그 | Role 기반으로 가도 감사로그가 없으면 "누가 처리했는지" 추적이 안 되는 문제가 그대로 남음. 신고처리/정지/환불 등은 별도 테이블에 기록해 추적 가능하게 함 |
| ADMIN 토큰 — Refresh 미발급 | 권한이 높은 계정일수록 토큰 탈취 시 피해가 커서, 노출 시간을 최소화하는 원칙을 적용. 일반 사용자(01-auth.md)와 의도적으로 다른 정책을 둔 이유를 면접에서 설명할 수 있는 포인트 |
| Redis 로그인 실패 카운터 재사용 | 관리자 로그인도 동일한 `/api/auth/login`을 쓰므로, 01-auth.md에 추가한 브루트포스 방지 로직이 별도 구현 없이 그대로 적용됨 |
