**배포**: Docker(WSL2) → Cloudflare(DNS) + Render(Spring Boot + PostgreSQL)

**관리자 페이지**: React 운영 화면(신고/차단·사용자·상품·결제 모니터링)은 기존 Spring Boot API를 그대로 사용. 통계 대시보드는 별도 Streamlit 앱으로, Kafka 통계 컨슈머(Python)가 적재한 집계 테이블을 조회해 시각화 — 다이어그램엔 단순화를 위해 별도 표기하지 않음 (07-admin.md 참고)

**결제 환경**: 토스페이먼츠 테스트 모드 사용 (실 결제 발생 없음). 운영 전환 시 사업자등록 기반 심사 후 라이브 키로 교체하는 구조로 설계.

# 7. 개발 프로세스 — AI 도구 활용 방식

"AI를 잘 쓴다"는 막연한 주장보다, AI 제안을 어떻게 검증하고 판단해서 채택/반려했는지를 기록으로 남기는 것을 차별점으로 삼는다.

## 7.1 Claude Code 기반 개발

- 프로젝트/사용자 스코프 CLAUDE.md를 작성해 코딩 컨벤션, 아키텍처 제약(모듈러 모놀리식 유지, 이벤트 기반 설계 원칙 등), 금지 패턴을 AI에게 명시적으로 전달
- Git 브랜치 전략(main → develop → personal → feature) 안에서 기능 단위로 Claude Code와 페어 프로그래밍
- AI 제안을 그대로 받아들이지 않고, 검토 후 수정/반려한 사례를 `docs/TROUBLE_SHOOTING.md`에 기록 (예: "Kafka 컨슈머 그룹 설계에서 AI는 단일 그룹을 제안했으나, 알림/색인이 독립적으로 재처리되어야 해 그룹을 분리하도록 직접 수정")
- 4.7(nfr.md)의 검증 작업(k6 부하테스트 스크립트, docker-compose 멀티 인스턴스 구성 등) 작성/실행에도 Claude Code를 보조 도구로 활용

## 7.2 디자인 단계 — Claude를 활용한 UI 목업

- React 프론트엔드 구현 전, Claude로 화면별 UI 목업/와이어프레임(상품 등록, 채팅, 결제, 마이페이지 등)을 먼저 빠르게 탐색
- 컬러/타이포그래피/레이아웃 같은 디자인 토큰을 먼저 정의한 뒤, 실제 컴포넌트 구현 단계에서 일관되게 적용
- 디자인 → 구현으로 넘어가기 전에 컴포넌트 구조를 미리 점검해, 구현 중 발생하는 리워크를 최소화

## 7.3 검증 원칙

- AI가 생성한 코드/설계는 그대로 채택하지 않고, 본인이 직접 동작 확인 + 기존 아키텍처 결정과의 일치 여부를 검토
- 검증 체크리스트: ① 동작 테스트 직접 실행 ② 성능/보안 영향 확인 (예: N+1 발생 여부) ③ 기존 설계 원칙(모듈러 모놀리식, 책임 분리 등)과의 일치 여부
- Redis/Kafka/Elasticsearch 등 AI 도구로 빠르게 붙인 인프라라도, 각 기술이 내부적으로 어떻게 동작하는지(예: Kafka 파티션/컨슈머 그룹, Redis Pub/Sub 전달 보장 수준)는 직접 학습해 면접에서 구두로 설명 가능한 수준을 유지
