# 3.1 회원/인증

- 회원가입 / 로그인 (이메일 + 비밀번호, BCrypt 해싱)
- JWT(Access Token) 발급 — 유효기간 20분
- 인증된 사용자만 API 접근 가능 (Spring Security 필터 기반)
- 본인 리소스만 수정/삭제 가능 (서비스 레이어에서 소유자 직접 비교)
- Refresh Token 발급 및 회전(rotation) — Redis에 저장(TTL 14일), HttpOnly 쿠키로 전달. 사용할 때마다 재발급하고 이전 토큰은 즉시 무효화, 이미 무효화된 토큰이 재사용되면 탈취로 간주해 해당 사용자의 전체 세션을 무효화 **(신규)**
- 로그아웃 시 토큰 무효화 (Refresh Token 삭제 + Access Token 블랙리스트) **(신규)**
- 인증 관련 로그에서 Authorization 헤더·토큰 원문 마스킹/제거 **(신규)**
- 인증·검증 예외 처리 표준화 — @Valid 실패, 토큰 사용자 미존재 등 미처리 예외를 공통 응답 포맷(ApiResponse)으로 통일 **(신규)**
- 비밀번호 정책 강화 (최소 길이/복잡도 검증) **(신규)**
- 소셜 로그인 (카카오 OAuth2) **(신규)**
- 로그인 실패 누적 시 일시적 지연/잠금 처리 (브루트포스 방지) **(신규)**
- 비밀번호 찾기/재설정 (이메일 인증 링크) **(신규)**
- 회원정보 수정 — 닉네임, 프로필 사진(Supabase Storage 재사용), 비밀번호 변경 **(신규)**
- 회원 탈퇴 **(신규)**
- 사용자 신고 / 차단 **(신규)**
- 사용자 Role(USER/ADMIN) 도입 — 관리자 인가의 기반, `07-admin.md`에서 사용 **(신규)**
- 회원가입 API는 요청 바디로 role을 받지 않고 항상 USER로 고정 — 권한 상승 방지 **(신규)**
- GET /api/users/me — 현재 로그인 사용자 정보/role 조회 API **(신규)**
- 동네(생활권) 인증 — GPS 좌표를 카카오맵 API로 행정동 전체 경로(시/도-시군구-동)로 변환해, 등록하려는 동의 경로와 정확히 일치하면 인증 처리. 1인당 1개 동네. 상품 조회 등 둘러보기는 비인증 사용자도 가능하고, **상품 등록·채팅 시작 같은 거래 행위에만 인증을 요구** **(신규)**

## 동네 인증 데이터 모델

UserNeighborhood(userId, 동코드, 인증 시점 GPS 좌표, 인증일시) 테이블을 신규로 추가. 동 마스터 데이터(동코드/동이름/상위 행정구역)는 공공데이터포털 자료로 시드 — 카카오맵 API가 행정동 경계를 기준으로 판정해주므로 동 중심좌표나 거리 계산은 불필요. GPS 스푸핑(위치 조작) 탐지는 포트폴리오 스코프에서는 인지하되 구현하지 않음(스코프 아웃)으로 명시.

## 기술/설계 선택 이유 (신규)

| 항목 | 선택 이유 |
| --- | --- |
| 카카오 OAuth2 (소셜 로그인) | 자체 비밀번호 관리 부담을 줄이고, 한국 사용자 대상 서비스에서 가장 보편적인 가입 경로를 제공. Spring Security OAuth2 Client로 표준 흐름(인가코드 → 액세스토큰 → 사용자정보조회)을 구현해 동작 원리를 직접 설명할 수 있는 학습 포인트로 활용 |
| Redis 기반 로그인 실패 카운터 | DB에 실패 횟수를 매번 기록하면 시도마다 트랜잭션 부하가 생기고 만료(TTL) 처리도 직접 구현해야 함. Redis는 키 TTL을 기본 지원해 "일정 시간 내 N회 실패 시 잠금" 정책을 키 만료만으로 단순하게 구현 가능 |
| Redis 기반 Refresh Token + 재사용 탐지 | DB 저장은 TTL 만료를 직접 구현해야 하지만, Redis는 키 TTL을 기본 지원해 만료 처리가 단순함. 토큰 회전(사용할 때마다 재발급)과 결합해, 무효화된 토큰이 다시 쓰이면 탈취로 간주하고 세션을 강제 종료하는 재사용 탐지(refresh token reuse detection)까지 구현 |
| 신고/차단 — 누적 임계치 기반 처리 | 신고 1건만으로 즉시 제재하면 오신고·악용 위험이 있음. 누적 신고 수가 임계치를 넘으면 운영자 검토 대상으로 플래그하는 단계적 처리로 설계 |
| 회원가입 시 role 강제 고정 | 클라이언트가 role을 직접 지정할 수 있게 두면 누구나 ADMIN으로 가입해버리는 권한 상승 취약점이 생김. 가입 경로에서는 항상 USER만 부여하고, ADMIN은 API로 노출하지 않음 |
| GET /api/users/me (role 노출용 조회 API) | Streamlit이 관리자 화면을 보여줄지 판단해야 하는데, JWT가 HS256 대칭키라 Streamlit이 직접 디코드하려면 서명 시크릿을 Python 환경에도 복제해야 함. 검증은 Spring Boot 한 곳에서만 하고 결과(role)만 내려줘 시크릿 노출 표면을 늘리지 않음 |
| 카카오맵 API (좌표 → 행정동 변환) | 자체적으로 행정동 경계 데이터(GeoJSON)를 구해 Point-in-Polygon을 구현하는 것보다, 이미 정확한 행정동 경계로 판정해주는 외부 API를 쓰는 게 정확도와 구현 비용 모두에서 유리함. 좌표→행정동이 이미 나오므로 별도 거리 계산(Haversine)이나 동 중심좌표 보관이 불필요해짐 |
| 동 전체 경로(시/도-시군구-동) 일치 비교 | 동 이름만 비교하면 동명이인 동네(예: "역삼동"이 여러 지역에 존재)를 혼동할 수 있음. 상위 행정구역까지 포함한 전체 경로로 비교해야 정확하게 동일 지역임을 판정할 수 있음 |
| 약한 접근 제한 (조회는 자유, 거래 행위만 인증 요구) | 조회까지 막으면 신규 유입 장벽이 높아짐. 실제 당근마켓도 등록/채팅 등 실제 거래 행위에만 동네 인증을 요구하는 방식을 채택 |
