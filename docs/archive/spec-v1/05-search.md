# 3.5 검색 및 추천 — 전체 신규 (신규)

## 문제 상황

기존 검색은 상품명 기반 단순 매칭(LIKE 검색)이라 표현 변형이나 오타를 잡지 못하고, 상품 수가 많아지면 풀스캔으로 느려진다. 상품 상세 페이지에서 "이런 상품도 있어요" 같은 추천이 없어 둘러보기 경험이 단방향으로 끝난다.

## 해결

Elasticsearch로 검색 전용 인덱스를 분리해 형태소 분석/오타 허용 검색을 지원하고, 상품 등록 시 생성된 임베딩(AI 에이전트 기능에서 만든 벡터, Supabase pgvector)을 활용해 "비슷한 상품" 추천을 함께 제공한다.

## 기능 상세

- 상품 등록/수정 시 Elasticsearch 인덱스에 동기화 (Kafka 이벤트로 비동기 색인)
- 한글 형태소 분석기(Nori) 기반 검색 — 띄어쓰기/오타 허용
- 자동완성 (검색어 입력 중 추천 키워드 표시)
- 인기 검색어 집계
- 상품 상세 페이지에서 "비슷한 상품" 추천 (Supabase pgvector 코사인 유사도 기반)
- 검색 결과 정렬/필터 (가격, 카테고리, 거래상태)
- 색인 실패 시 재시도/DLQ 처리 — 최대 3회 지수 백오프(exponential backoff) 재시도 후에도 실패하면 product.index.dlq 토픽으로 이동, 핵심 트랜잭션(상품 등록 자체)은 영향받지 않음

## 기술 선택 이유

| 기술 | 선택 이유 |
| --- | --- |
| Elasticsearch + Nori 분석기 | MySQL LIKE 검색은 인덱스를 못 타 데이터가 많아지면 풀스캔이 발생하고 형태소 분석이 안 됨. 역색인 구조로 대량 텍스트 검색에 최적화되어 있고, Nori로 한글 형태소 단위 검색을 지원해 검색 품질을 높임 |
| Kafka 기반 비동기 색인 동기화 | 상품 등록 시 MySQL 저장과 ES 색인을 동기로 처리하면 ES 장애가 상품 등록 자체를 실패시킬 수 있음. 이벤트 발행 후 별도 컨슈머가 색인하게 분리해 핵심 트랜잭션을 검색 인프라 장애로부터 격리 |
| Supabase pgvector 기반 추천 | AI 상품 등록 도우미에서 이미 만든 임베딩을 재사용해, 텍스트 매칭만으로 못 찾는 의미적 유사 상품(예: "축구화"↔"풋살화")을 잡아냄. 별도 벡터 DB 없이 SQL로 유사도 검색 가능 |
| 자동완성 (ES Completion Suggester) | 검색어 입력 중 후보를 보여주면 검색 성공률과 UX가 향상됨 |
| 재시도 + DLQ (DefaultErrorHandler + DeadLetterPublishingRecoverer) | 색인 실패를 무한 재시도하면 컨슈머가 멈추고, 그냥 버리면 검색 결과가 누락됨. Spring Kafka가 기본 제공하는 에러 핸들러로 N회 재시도 후 DLQ로 격리해, 직접 재시도 로직을 짤 필요 없이 표준화된 방식으로 처리 |

**검색 인프라 역할 분리**: 텍스트 전문 검색(오타/형태소)은 Elasticsearch, 의미 기반 유사도 검색은 Supabase pgvector로 역할을 명확히 나눠 사용한다.
